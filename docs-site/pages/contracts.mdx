## What each contract does

### LitZapRegistry
Maps a human `name.zap` handle to a wallet address. Names are `[a-z0-9_]`, 3–20 characters, unique, one per address. Anyone — including other LitVM apps — can resolve a name on-chain.

### LitZapPay
Non-custodial payments: funds pass straight through from sender to recipient (native or ERC-20). Also emits payment **requests** and supports **boomerang claims** (send-to-claim with auto-return). The protocol fee is hard-capped in the contract and currently set to **zero**.

### LitZapEscrow
The contract behind **pay-by-social**. A sender locks funds against an off-chain identity (e.g. an X handle), and the funds are released only to the verified owner — or refunded to the sender after expiry. Designed as a public, composable primitive. Full details in [Pay-by-social](/pay-by-social).

### LitZapSubscriptions
Recurring payments via on-chain allowances — the basis for memberships and scheduled payouts.

### LitZapCapsule
ERC-721 collectibles for receipts, badges, and moments — an extensible on-chain object LitZap features can build on.

## Safety properties

- **Non-custodial** — completed payments never sit in a LitZap contract; the only held funds are open escrows/claims, which are always refundable by the sender after expiry.
- **Fee cap** — the maximum protocol fee is fixed in code and cannot be raised past its ceiling.
- **Guarded** — payment and escrow contracts use reentrancy protection and pausability.

See [Security &amp; trust](/security) for the full model.
